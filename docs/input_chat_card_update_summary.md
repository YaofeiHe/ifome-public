# 输入框、聊天框与卡片更新摘要

这份文档只总结最近围绕“统一输入”和“卡片更新”做的改动，适合快速回顾。

## 改了什么

### 1. 输入页变成统一消息框

输入页现在不再区分“文本输入”“图片 OCR”“文件导入”这些分散入口，而是统一成一个聊天框：

- 可以直接贴文本
- 可以直接贴链接
- 可以在框里写本地文件路径
- 可以直接粘贴剪贴板里的图片或文件
- 可以点击按钮上传图片、PDF、文本文件

前端文件：

- `apps/web/app/ingest/page.tsx`

输入页现在还维护本地多会话历史：

- 一个输入主题可以连续提交多轮输入
- 当前历史会从第一轮到最后一轮完整展示输入和输出
- 页面底部有可滚动历史列表，摘要最多两行
- 可以创建新历史、切换历史、删除单个历史或多选删除
- 输入页仍然保持“输入 + 输出”的工作流形态，不强行套聊天气泡

后端入口：

- `apps/api/src/main.py` -> `ingest_unified`
- `apps/api/src/services.py` -> `ingest_unified`

### 2. 聊天页也对齐成同样的输入方式

聊天页现在和输入页的附件接入体验一致，同样支持：

- 文本
- 链接
- 本地路径
- 上传文件
- 粘贴图片和文件

前端文件：

- `apps/web/app/chat/page.tsx`

聊天控制台的会话历史语义也同步收口：

- 当前对话完整展示从第一轮到最后一轮的历史
- 其他对话在页面底部历史列表里滚动展示
- 历史摘要最多两行，旧会话继续按压缩摘要策略降低重启后的恢复成本
- Boss 回复模拟保持社交软件式对话框：Boss / HR 消息在左，建议回复在右
- 修改提示词固定在建议回复下方，回复策略和分析改为点击展开

后端入口：

- `apps/api/src/main.py` -> `/chat/query-unified`
- `apps/api/src/services.py` -> `_collect_chat_attachments`

### 3. 所有文件先转文本

现在文件不会直接走各自独立的业务分支，而是先统一转文本：

- 图片：OCR
- PDF：正文提取，必要时 OCR fallback
- 文本文件：直接抽文本

OCR 顺序：

- 独立 OCR LLM
- 通用 LLM OCR
- 本地 `tesseract`

核心文件：

- `apps/api/src/services.py`
- `core/tools/ocr_client.py`

### 4. 输入页的目标是写卡或更新卡

输入页走的是信息收集链路，目标是：

- 新建卡片
- 更新已有卡片
- 或把新旧卡重组成大卡-小卡结构

主流程：

1. `ingest_unified`
2. `ingest_auto`
3. section 切分 / 岗位拆分
4. `classify`
5. `extract`
6. `match_profile`
7. `score`
8. `reminder_plan`
9. `persist`
10. `render`

### 5. 聊天页的目标是回答，不是默认建卡

聊天页虽然也支持文件和图片，但默认用途是：

- 附件转文本
- 进入 RAG 检索
- 生成回答、引用、query plan

默认不会直接生成卡片。

例外是：

- 如果附件看起来像简历
- 并且开启画像自动更新

那它会优先更新记忆，而不是建卡。

### 6. 去重和卡片更新不是简单覆盖

`persist` 现在做的是“规则优先 + 按需 LLM”：

- 先用链接、apply_url、公司、岗位、文本相似度和 SQLite 历史卡片做候选判断
- 如果是同公司/相似岗位，直接触发 LLM 判断
- 结果可能是：
  - 更新旧卡
  - 新建卡
  - 或触发 regroup

核心文件：

- `core/graph/nodes/persist.py`

### 7. 大卡-小卡现在有两种建立方式

#### 显式建立

如果一次输入本身就是“一个公司/业务线下面多个岗位”，系统会：

- 先拆多个 role 子卡
- 再生成一个 overview 父卡

#### 动态 regroup

如果旧卡原本没进大卡-小卡框架，但新卡进来后发现它们属于同一组：

- 先做关系判断
- 再扩成关系连通簇
- 最后重建 overview + role 子卡结构

这解决的是：

- “多个旧卡其实互相关联，但之前没组织起来”

### 8. 图片导入现在也能进入这套更新链

以前图片导入更像独立 OCR 支路，现在已经改成：

- 先 OCR
- 再把 OCR 文本送回 `ingest_auto`

所以图片导入也能正常触发：

- 分类
- 抽取
- 去重
- 旧卡更新
- 大卡-小卡 regroup

### 9. 子卡显示现在更稳了

父卡展开子卡时，前端现在优先使用父卡自己的 `job_child_item_ids`，不再只依赖前端反向扫描 `job_parent_item_id`。

这修掉了：

- 后端明明已经挂好了多个子卡
- 但前端展开时只显示其中一部分

核心文件：

- `apps/web/app/items/page.tsx`

市场大卡的小卡预览也做了重复标题兜底：

- 后端回填 preview 时会去重
- 前端渲染 key 不再只使用标题文本
- 父卡 child id 也会去重，避免同一子卡被显式挂载和反向推断各渲染一次
- 避免同名标题或同一 `item_id` 触发 React duplicate key 报错

### 11. LLM 调用统一带当前时间戳

通用 LLM gateway 现在会在请求模型前自动补充当前时间：

- system prompt 头部加入 `当前时间戳`
- user payload 加入 `current_timestamp`
- OCR LLM prompt 也带同样时间戳

这样所有涉及“今天 / 最近 / DDL / 近 24 小时”的分析都有明确时间基准。

### 12. 市场标题排序改成 LLM + 关键词融合

市场刷新现在先抓标题列表，再决定抓哪些正文：

- 机器之心优先从 `article_library` API 获取近 24 小时标题
- 机器之心正文抓取优先从 `article_library/articles/<slug>.json` 获取完整 HTML 正文
- 标题先按用户画像关键词、市场价值词和探索价值打规则分
- 如果 live LLM 可用，再用 `market_watch_title_rank` 做标题级评分
- prompt 会带入用户画像词、关键词分、市场价值分，以及历史卡片 / 向量召回参考
- 最终排行融合 LLM 分和关键词分，选出设置里配置数量的标题再抓正文
- 小脚本支持 `--fetch-top N`，可直接验证“标题列表 + 前 N 篇正文”两段链路

### 10. 源文件也进入了统一管理

原始输入现在会归档到本地：

- 文本会存成 `.txt`
- 图片/PDF/原文件会保留格式
- 卡片上可以“访问源文件”
- 设置页可以配置清理策略

相关能力：

- 源文件访问
- 清理规则
- 卡片删除联动删除失去引用的源文件

## 最终理解

这次改造后的核心思路是：

- 统一输入体验
- 文件先转文本
- 文本再走统一分析工作流
- 输入页负责信息收集和卡片更新
- 聊天页负责问答和记忆辅助
- 卡片更新不只是“新增”，而是“新增 / 更新 / regroup”

- `docs/ifome_storage_rag_notes.md`
