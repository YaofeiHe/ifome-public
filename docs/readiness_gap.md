# Readiness Gap

这份文档只讨论一件事：

- 在 `Phase 14` 完成之后，项目离“更稳定、更像真实产品”还差哪些工程项

它不再重复记录已经完成的阶段，已经完成的内容统一收敛到：

- `docs/next_execution_plan.md`

## 当前已经具备

当前已经完成并验证过的能力：

- 单 Agent 显式工作流
- 本地前后端真实运行
- live `classify / extract`
- link 正文抓取
- image 上传 + OCR
- SQLite 持久化
- Boss 直聘最小 connector
- grounded chat / RAG
- DashScope embedding + Qdrant 真实联调

所以现在的 gap，不再是“能不能跑起来”，而是“怎么继续把它做扎实”。

## P0：下一阶段最值得尽快补的项

### 1. 去重与幂等

- 同一个链接、同一段 JD、同一张截图重复录入时，应该做去重或版本合并
- 这是从 demo 向日常使用迈进时最先会遇到的问题

### 2. 检索质量增强

- 当前 RAG 已可用，但 rerank 还是 heuristic
- 还缺 retrieval evaluation、召回对比和失败样本回放

### 3. Boss 直聘真实读取链路

- 现在只完成了最小 payload 接入
- 还没有真实登录态、抓取、导出或同步链路

## P1：为了“更连续使用”需要补的项

### 4. 中文 OCR 增强

- 当前 OCR 路径已经打通
- 但中文截图识别效果仍可以继续增强

### 5. 网页正文抽取增强

- 当前是轻量抓取方案
- 对复杂页面的正文质量还可以提升

### 6. 通知真正发送

- 当前 reminder queue 已存在
- 但邮件、日历、Web Push 还只是 adapter 预留

## P2：为了“更像产品”需要补的项

### 7. 分类后的二级抽取 schema

- 当前主要仍收敛在 `ExtractedJobSignal`
- 后续更适合拆成按类别的二级 schema

### 8. 排名层语义升级

- 当前 ranking 主要仍是规则打底
- 后续可以引入更强的语义匹配与解释生成

### 9. 基础鉴权

- `user_id / workspace_id` 已经预留
- 但还没有真正的登录、鉴权和权限边界

## P3：为了“长期维护”需要补的项

### 10. 日志与回放

- 当前 trace 主要还是开发态可见
- 后续需要更系统的请求级日志和失败回放

### 11. 部署与配置管理

- 环境隔离
- 密钥管理
- 数据备份
- 迁移脚本

### 12. 验证后的 runtime 替换

- 当前项目已经验证了 LangGraph 作为 POC 编排器的价值
- 下一阶段才是考虑逐步迁到自定义 runtime 的时机
- 这一步应该放在真实验证之后，而不是抢在验证前做

## 面试时怎么讲这些 gap

建议表达成：

- 这个项目已经完成了第一轮真实能力验证
- 当前最主要的 gap 不是“有没有框架”，而是“如何把输入质量、检索质量、连接器和长期可维护性继续做实”
- runtime 替换不是没想到，而是有意放在验证后的下一阶段
