# 求职信息流 Agent 项目白皮书

## 1. 项目概述

### 1.1 项目名称
**Job Hunt Agent / 求职信息流 Agent**

### 1.2 一句话定义
一个面向求职场景的个人信息流 Agent：接收招聘相关信息，完成结构化抽取、个性化优先级判断、提醒编排与后续问答支持，帮助用户降低信息处理成本并减少重要信息遗漏。

### 1.3 项目目标
本项目的直接目标不是构建一个“泛聊天机器人”，而是构建一个**高约束、可解释、可持续扩展的求职工作流 Agent 系统**。

项目同时服务于两类目标：

1. **真实使用目标**：帮助用户管理分散的求职信息，尤其是来自微信、聊天截图、招聘链接、文本通知等非结构化输入。
2. **工程训练目标**：作为一个可讲清架构、可展示 Agent 能力、可在面试中深入讨论的工程项目，用于训练和展示 AI Agent 开发能力。

### 1.4 当前目标用户
第一阶段以**单用户求职场景**为主，但从架构上预留多用户支持，不将任何核心逻辑写死为“仅面向某一位用户”。

默认用户画像示例：
- 主方向：后端 / 工程型算法 / 数据分析
- 重点方向：AI 大模型工程化应用、智能驾驶
- 重点价值：减少信息处理负担，避免错过重要 DDL、面试、宣讲和高匹配岗位

---

## 2. 问题定义

### 2.1 用户痛点
求职信息往往分散在多个入口：
- 微信聊天消息
- 招聘链接
- 聊天截图
- 公众号内容
- 后续可扩展到小红书、邮件、短信、Boss 直聘岗位内容与 HR 沟通记录

问题不在于“信息太少”，而在于：
- 信息过多，无法及时处理
- 非结构化内容难以快速判断是否重要
- DDL / 面试时间 / 地点容易遗漏
- 同类信息缺少统一整理与长期追踪
- 用户希望系统能根据自己的画像自动判断“值不值得投 / 要不要现在处理”

### 2.2 项目解决的问题
本项目重点解决以下问题：

1. **输入统一化**：把链接、截图、纯文本等输入转成统一对象。
2. **信息结构化**：自动提取公司、岗位、地点、时间、投递链接、DDL 等关键字段。
3. **个性化筛选**：结合长期记忆与当前求职状态判断匹配度和优先级。
4. **提醒编排**：把重要时间点放入提醒池，并通过邮件、日历、Web Push 等方式触达。
5. **后续行动支持**：基于已处理信息，支持“今天该投什么”“面试可能问什么”“项目怎么讲”等问答任务。

---

## 3. 产品定位

### 3.1 项目不是
- 不是一个纯聊天机器人
- 不是一个泛自动化平台
- 不是一个一开始就全自动监听微信的系统
- 不是一个高度自治、不可控的多 Agent 黑盒

### 3.2 项目是
- 一个**事件驱动的 Agent 工作流系统**
- 一个**面向求职场景的信息处理与决策支持平台**
- 一个**低成本、可解释、可迭代到手搓内核的工程项目**

### 3.3 核心设计原则
1. **从简单系统开始**：POC 阶段优先做单 Agent 工作流，而非多 Agent。citeturn140585search1turn140585search13
2. **高约束、低黑盒**：尽可能让状态、节点、输入、输出、模型路由都显式可见。
3. **优先保证可用性**：先解决“降低处理成本”和“减少遗漏”，再扩展复杂能力。
4. **分层设计**：输入层、工作流层、工具层、记忆层、展示层分离。
5. **框架先行、内核后移植**：POC 借助 LangGraph；成熟后迁移为手写 Runtime。

## 4. 需求范围

### 4.1 POC 必做能力

#### 输入
- 招聘链接
- 聊天截图
- 纯文本招聘消息

#### 输出
- 默认一行摘要
- 提取字段：公司 / 岗位 / 地点 / 时间 / 投递链接
- 高亮重要字段：DDL、笔面试地点、宣讲时间
- 自动进入提醒池
- 原始信息备份

#### 记忆
- 长期记忆：方向、技能、经历、偏好、目标行业
- 短期上下文：最近关注/投递/面试进展
- 上传时支持标记：
  - 仅当前任务使用
  - 写入长期记忆

#### 对话控制台
- 在已处理结果基础上继续追问
- 为岗位生成面试准备材料
- 生成当前应优先投递岗位列表
- 支持通过明确指令更新记忆

### 4.2 POC 暂不做
- 自动监听微信私聊
- 自动删除或修改邮件/短信/未接电话记录
- 自动投递
- 自动给 Boss 直聘 HR 发消息
- 复杂多 Agent 编排

## 5. 架构设计

### 5.1 总体架构

系统分为五层：

1. **接入层**：接收文本、链接、图片、未来外部连接器数据。
2. **Agent 工作流层**：完成分类、抽取、匹配、打分、提醒规划。
3. **记忆层**：管理长期画像、动态求职状态、文档资料。
4. **工具层**：OCR、网页抽取、邮件、日历、通知、搜索等。
5. **展示与交互层**：Web 卡片页 + 聊天控制台 + 管理页。

### 5.2 为什么 POC 采用 LangGraph
LangGraph 的定位是低层级 Agent 编排框架，重点在 durable execution、human-in-the-loop、streaming、stateful orchestration；并且官方明确指出 LangGraph 不强制依赖 LangChain，高级抽象可以后续再加。citeturn140585search0turn140585search12

对于本项目，这意味着：
- 适合定义清晰状态图
- 适合把每一步显式存档
- 适合在关键节点加入人工修正
- 适合作为未来手写 Runtime 的脚手架

### 5.3 为什么后续要手搓内核
POC 用框架是为了快速验证，但长期来看，以下理由支持迁移到手写 Runtime：
- 需要更清晰地控制状态迁移
- 需要更精细的模型路由与成本控制
- 需要更清楚的日志与可解释性
- 需要更好地向面试官展示“不是黑盒调用框架”

## 6. Agent 工作流设计

### 6.1 核心状态图
建议第一版固定为如下流程：

`ingest -> normalize -> classify -> extract -> match_profile -> score -> reminder_plan -> persist -> render`

### 6.2 节点职责

#### ingest
接收原始输入，分配 `trace_id`，记录来源、时间、用户、输入类型。

#### normalize
- 链接：抓取正文与元信息
- 图片：OCR 抽文本并做轻清洗
- 文本：清洗空白、表情、乱码、换行
- 统一为标准输入对象 `NormalizedItem`

#### classify
判断输入类型，例如：
- 岗位
- 面试通知
- 宣讲会
- 内推信息
- 泛资讯
- 噪声

#### extract
提取结构化字段：
- company
- role
- city
- interview_location
- ddl
- event_time
- apply_url
- source_url
- raw_text

#### match_profile
结合用户长期画像和当前求职状态，判断：
- 与主方向是否匹配
- 与重点行业是否匹配
- 是否需要优先关注
- 是否是“必须看”的岗位

#### score
输出：
- 匹配分
- 紧急度
- 建议动作（投递 / 收藏 / 稍后看 / 忽略）

#### reminder_plan
若识别到 DDL / 宣讲 / 笔试 / 面试时间，则生成提醒任务。

#### persist
保存原始内容、结构化字段、摘要、匹配结果、提醒任务、追踪状态。

#### render
生成前端卡片：
- 一行摘要
- 核心字段
- 高亮字段
- 优先级标签

## 7. 记忆系统设计

### 7.1 记忆分层
本项目不把“记忆”简单理解为一段拼接 prompt，而是显式分为三类：

#### A. Profile Memory
长期、相对稳定的用户画像，例如：
- 求职方向
- 技能标签
- 项目经历标签
- 行业偏好
- 城市偏好
- 明确排除项

#### B. Career State Memory
动态变化的求职状态，例如：
- 已投岗位
- 面试进展
- 重点关注公司
- 最近待处理事项
- 当前准备重点

#### C. Document Memory
用户上传的文档和资料，例如：
- 简历
- JD
- 面试题整理
- 项目介绍
- 复习笔记

### 7.2 “把本项目写入长期记忆”的临时功能
由于本项目本身也服务于 Agent 学习和面试准备，开发阶段可以允许把“项目自身信息”写入长期记忆，以便系统在后续生成面试准备内容时引用。

但该能力必须作为**临时、可拆卸的独立模块**设计，建议：
- 单独放在 `feature_flags` 或 `experimental` 域
- 与通用用户画像逻辑分离
- 不影响多用户通用版本
- 可以一键关闭或移除

## 8. 提醒系统设计

### 8.1 提醒池
提醒系统的最小单元不是“字段”，而是一个可调度任务：

- `reminder_id`
- `user_id`
- `source_item_id`
- `title`
- `trigger_time`
- `priority`
- `channel`
- `status`

### 8.2 触达渠道
POC 优先推荐：
- 邮件提醒
- 日历事件

Google Calendar API 官方支持通过 `events.insert()` 创建事件，最少需要 `start` 和 `end` 信息。citeturn140585search2turn140585search22

Web 方向的系统通知和后台触达可作为第二阶段能力：
- Notifications API 可显示系统级通知。citeturn140585search11turn140585search15
- Push API 允许服务端把推送发送给浏览器中的 Service Worker。citeturn140585search3turn140585search7

### 8.3 提醒优先级
建议将提醒优先级分成：
- P0：今日内到期 / 面试即将开始
- P1：3 天内到期 / 宣讲会 / 笔试
- P2：一周内建议处理
- P3：长期关注项

## 9. 成本控制与模型路由

### 9.1 原则
由于高推理模型 API 成本高、额度有限，本项目应采用**分层模型路由**。

### 9.2 低成本模型负责
- 分类
- 基础字段抽取
- OCR 后文本修正
- 一行摘要
- 去重
- 模板化提醒文案
- 常规匹配打分

### 9.3 高推理模型负责
- 模糊岗位是否值得投
- 多岗位优先级排序
- 面试问题生成
- 项目介绍组织
- 冲突信息消解
- 深层匹配解释

### 9.4 升级策略
只有在以下条件下才升级到高推理模型：
- 信息不完整且影响决策
- 识别到高优先级岗位
- 用户主动要求深度分析
- 需要面试准备、项目讲述或投递策略建议

## 10. 数据存储与归档策略

### 10.1 原始信息备份
系统必须保留原始信息，便于：
- 回溯错误
- 重跑抽取逻辑
- 后续模型升级后重新分析

### 10.2 分层归档
#### 热数据
最近 30~60 天：保留全文、图片、结构化结果和可重算信息。

#### 温数据
保留摘要、关键字段、来源链接、处理结论。

#### 冷数据
主库仅保留索引、摘要和引用，原始内容迁移到对象存储或归档目录。

## 11. 多用户与产品化预留

虽然第一阶段以单用户场景为主，但数据层必须预留：
- `user_id`
- `workspace_id`
- `memory_scope`
- `connector_id`
- `privacy_level`

避免出现以下反模式：
- 全局只有一份简历和画像
- 配置文件硬编码某个人信息
- 所有提醒默认发往唯一邮箱
- 记忆写入逻辑与单用户数据结构耦合

## 13. 面向面试的工程要求

为了确保项目可用于面试交流，必须遵守以下要求：

### 13.1 避免黑盒
- 每个节点都有明确输入输出
- 每个阶段有中间产物落盘
- 模型路由有显式规则
- 重要工具调用有独立封装

### 13.2 文档完整
项目应至少包含：
- PRD
- 架构文档
- 记忆设计文档
- 提醒设计文档
- 未来规划文档
- Demo 脚本
- 面试提纲文档

### 13.3 代码可讲清
- 关键类必须有 docstring
- 节点函数必须说明职责、输入、输出、失败处理
- 不允许把复杂逻辑全部塞进单个 prompt
- 不允许无说明地串联多个外部库调用

### 13.4 可观测性
- 每条任务必须有 `trace_id`
- 记录耗时、模型名、成本估算、失败原因
- 支持样本回放

## 14. 技术选型建议

### 14.1 POC 推荐
- **后端**：FastAPI
- **工作流**：LangGraph
- **模型接入**：兼容 OpenAI-style API
- **数据库**：SQLite 起步，后续迁移 PostgreSQL
- **前端**：Next.js / React（或更轻量的 Web 前端）
- **OCR**：可插拔 OCR 模块
- **异步任务**：RQ / Celery / 简化任务队列

### 14.2 迁移方向
- 保留外层接口
- 抽象状态对象、节点接口、工具分发器
- 用手写 Runtime 逐步替换 LangGraph 执行层

## 16. 风险与应对

### 风险 1：过度追求全自动
应对：优先做“手动转发 + 自动整理”，先保证链路可用。

### 风险 2：Agent 过黑盒
应对：显式状态图、独立工具封装、中间结果落盘、日志与回放。

### 风险 3：模型成本失控
应对：低成本模型默认处理，高推理模型按规则升级。

### 风险 4：后续扩展耦合严重
应对：记忆、工具、连接器、提醒系统全部模块化。

### 风险 5：项目只能服务单用户
应对：提前在数据模型与配置层预留 `user_id` 和 `workspace_id`。

---

## 17. 成功标准

如果 POC 满足以下条件，即可视为成功：
- 能稳定接收三类输入：链接、截图、纯文本
- 能生成清晰的一行摘要与关键字段
- 能对 DDL / 地点 / 时间进行高亮提醒
- 能结合画像做基础优先级判断
- 能基于已有记录继续做岗位建议和面试准备
- 代码、文档、架构都可在面试中讲清楚
